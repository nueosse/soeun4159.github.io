Git 배우기: 커밋부터 브랜치, 리베이스까지

Git의 기본 흐름과 주요 명령어를 정리한 학습 기록입니다. commit, branch, checkout, merge, rebase, reset, revert, cherry-pick처럼 자주 쓰이는 명령어를 중심으로 Git의 트리 구조와 협업 흐름을 이해하는 데 초점을 두었습니다.

개요

Git은 프로젝트의 변경 이력을 기록하고 관리하는 버전 관리 시스템이다. 이번 학습에서는 단순히 명령어 사용법을 익히는 데서 그치지 않고, commit이 어떻게 연결되고 branch가 어떤 방식으로 이동하는지 Git의 내부 흐름을 이해하는 데 집중했다.

Learn Git Branching 환경을 활용해 commit 그래프의 변화를 직접 확인하면서 merge, rebase, reset, cherry-pick, push, pull 같은 주요 명령어가 실제로 어떤 변화를 만드는지 실습 중심으로 학습했다.

학습 Objective

이번 Git 학습의 핵심 목표는 단순히 명령어를 외우는 것이 아니라, Git이 commit과 branch를 기반으로 어떻게 동작하는지 구조적으로 이해하는 것이었다. 각 명령어를 실행했을 때 commit 그래프가 어떻게 변하는지 직접 확인하면서 Git의 흐름을 자연스럽게 익히는 데 집중했다. 

핵심 목표
- commit과 branch의 구조 이해하기
- HEAD와 상대 참조를 통해 현재 위치 정확히 파악하기
- merge와 rebase의 동작 방식과 차이 이해하기
- reset / revert로 작업 되돌리기 익히기
- cherry-pick과 interactive rebase로 commit 다루기
- push / pull / fetch를 통한 원격 저장소 협업 흐름 이해하기
1. Git 기본

Git 기본

Git 기본 단계에서는 commit, branch, checkout, merge, rebase처럼 Git을 사용할 때 가장 먼저 익혀야 하는 명령어들을 배웠다. 이 단계의 핵심은 commit이 쌓이는 방식과 branch가 특정 commit을 가리키는 포인터라는 점을 이해하는 것이다.

commit

git commit은 현재 작업 내용을 하나의 기록으로 저장하는 명령어이다. Git에서 commit은 프로젝트의 특정 시점을 저장하는 스냅샷 역할을 한다. 

git commit

branch

git branch는 새로운 작업 흐름을 만들 때 사용한다. branch는 commit을 복사하는 것이 아니라 특정 commit을 가리키는 이름표에 가깝다. 

git branch newImage

checkout

git checkout은 이동 명령어이다. 특정 branch나 commit으로 이동할 때 사용하며, -b 옵션을 사용하면 branch 생성과 이동을 한 번에 할 수 있다. 

git checkout main
git checkout newImage
git checkout -b bugFix

merge

git merge는 다른 branch의 작업 내용을 현재 branch로 합치는 명령어이다. 두 branch의 변경 이력을 보존하면서 합쳐지기 때문에 merge commit이 생길 수 있다. 

git checkout main
git merge bugFix

rebase

git rebase는 현재 branch의 commit들을 다른 branch의 최신 commit 뒤로 다시 쌓는 명령어이다. merge보다 히스토리를 더 깔끔하게 만들 수 있다. 

git checkout bugFix
git rebase main
2. 다음 단계로

다음 단계로

다음 단계에서는 HEAD, 상대 참조, branch 강제 이동, 작업 되돌리기를 배웠다. 이 부분은 현재 내가 Git 트리의 어디에 있는지 정확히 파악하는 데 중요하다.

HEAD와 Detached HEAD

HEAD는 현재 내가 보고 있는 위치를 의미한다. 보통은 branch를 가리키지만, 특정 commit으로 직접 이동하면 HEAD가 branch에서 분리된 Detached HEAD 상태가 된다. 

git checkout C1

상대 참조

^는 부모 commit 하나를 가리키고, ~<num>은 여러 단계 이전 commit을 가리킨다. 

git checkout main^
git checkout HEAD~4

branch 강제 이동

git branch -f를 사용하면 branch가 가리키는 위치를 원하는 commit으로 강제로 옮길 수 있다. 

git branch -f main HEAD~3

reset과 revert

git reset은 branch가 가리키는 commit 위치를 이전으로 옮겨 local 작업을 되돌릴 때 사용하고, git revert는 기존 commit을 삭제하지 않고 되돌리는 commit을 새로 만들어 공유된 작업을 안전하게 되돌릴 때 사용한다. 

git reset HEAD~1
git revert HEAD
3. 코드 이리저리 옮기기

코드 이리저리 옮기기

이 단계에서는 원하는 commit만 골라 가져오거나, commit 순서를 바꾸는 방법을 배웠다. 전체 branch를 합치지 않고 필요한 변경 사항만 가져올 때 유용하다.

cherry-pick

git cherry-pick은 다른 branch에 있는 특정 commit만 현재 branch로 가져오는 명령어이다. 

git cherry-pick C3 C4 C7

interactive rebase

git rebase -i는 여러 commit을 대화형으로 편집하는 명령어이다. commit 순서를 바꾸거나, 필요 없는 commit을 제거하거나, 여러 commit을 합치는 등 히스토리를 정리할 수 있다. 

git rebase -i HEAD~4
4. 종합선물세트

종합선물세트

종합선물세트 단계에서는 cherry-pick, rebase -i, amend, tag, describe처럼 commit을 더 세밀하게 다루는 명령어들을 함께 정리했다.

amend

git commit --amend는 가장 최근 commit을 정정할 때 사용한다. 직전 commit 메시지나 내용을 수정할 때 유용하다. 

git commit --amend

tag

git tag는 특정 commit에 이름표를 붙이는 명령어이다. 주로 버전 표시나 중요한 지점을 표시할 때 사용한다. 

git tag v1 C1

describe

git describe는 특정 commit이 가장 가까운 tag로부터 얼마나 떨어져 있는지 알려준다. 결과는 <tag>-<numCommits>-g<hash> 형태로 나타난다. 

git describe main

# 예시
v1-2-gC2
5. 고급 문제

고급 문제

고급 문제에서는 여러 branch를 연속으로 rebase하거나, merge commit처럼 부모가 여러 개인 commit을 다루는 방법을 배웠다. 이 단계부터는 commit 그래프를 눈으로 보면서 명령어 결과를 예측하는 연습이 중요하다.

연속 rebase

여러 branch가 있을 때 기준 branch 위로 차례대로 rebase하여 전체 흐름을 정리할 수 있다. git rebase main bugFix처럼 쓰면 bugFix branch를 main 위로 옮긴다는 뜻이다. 

git rebase main bugFix
git rebase bugFix side
git rebase side another

다수의 부모

merge commit은 부모 commit을 여러 개 가질 수 있다. 이때 ^, ^2, ~를 조합해서 원하는 위치로 이동할 수 있다. 

git checkout main^
git checkout main^2
git checkout HEAD~^2~2
6. Push & Pull -- Git 원격저장소!

Push & Pull -- Git 원격저장소!

이 단계에서는 원격 저장소의 기본 흐름을 배웠다. 원격 저장소는 협업을 위해 사용하며, 로컬 저장소와 원격 저장소의 branch는 서로 따로 존재한다.

clone

git clone은 원격 저장소를 로컬 환경으로 복사하는 명령어이다. 

git clone <repository-url>

remote branch

원격 branch는 학습 환경에서는 o/main, 실제 Git에서는 보통 origin/main처럼 표시된다.

fetch

git fetch는 원격 저장소의 최신 정보를 가져오지만, 현재 branch에 바로 합치지는 않는다. 

git fetch

pull

git pull은 기본적으로 git fetch와 git merge를 합친 명령어이다. 현재 branch가 origin/main을 추적하고 있다면, 원격 변경 사항을 가져온 뒤 현재 branch에 병합한다. 

git pull

# 현재 branch가 origin/main을 추적하는 경우의 흐름
git fetch
git merge origin/main

push

git push는 내 로컬 commit을 원격 저장소에 업로드하는 명령어이다. 

git push

엇갈린 히스토리

내가 작업하는 동안 다른 사람이 원격 저장소에 먼저 commit을 올리면 로컬과 원격의 히스토리가 엇갈릴 수 있다. 이때는 fetch 후 rebase 또는 merge로 원격 변경 사항을 먼저 반영한 뒤 push한다. 

git fetch
git rebase origin/main
git push

# 또는
git pull --rebase
git push

잠겨버린 main branch

실제 협업에서는 main branch에 직접 push하지 못하도록 막아두는 경우가 많다. 이때는 feature branch를 만들어 작업한 뒤 pull request를 통해 병합한다. 

git checkout -b feature
# 작업 후 commit
# feature branch를 push하고 pull request 생성
7. "origin" 그 너머로 -- 고급 원격 저장소

"origin" 그 너머로 -- 고급 원격 저장소

마지막 단계에서는 원격 저장소를 더 세밀하게 다루는 방법을 배웠다. push, fetch, pull에 인자를 직접 주면 원하는 branch나 commit을 선택해서 주고받을 수 있다.

pull --rebase

git pull --rebase는 원격 변경 사항을 가져온 뒤, 내 로컬 commit을 그 위에 다시 쌓는 방식이다. 즉 fetch 후 merge가 아니라 fetch 후 rebase를 수행한다. 

git pull --rebase
git push

원격 작업과 merge하기

원격 저장소의 내용을 가져온 뒤 여러 branch를 merge로 합칠 수도 있다. 이 방식은 branch가 합쳐진 기록을 명확하게 남긴다. 

git checkout main
git pull
git merge side1
git merge side2
git merge side3
git push

원격 저장소 추적하기

로컬 branch가 어떤 원격 branch를 따라갈지 정하는 것을 upstream 설정이라고 한다. 

git checkout -b foo origin/main
git branch -u origin/main foo

push의 인자

git push는 <source>:<destination> 형태로 source와 destination을 직접 지정할 수 있다. source는 로컬에서 보낼 branch나 commit이고, destination은 원격 저장소에서 갱신할 branch이다. 

git push origin main
git push origin foo
git push origin foo:main
git push origin main:newBranch
git push origin main^:foo

fetch의 인자

git fetch도 특정 branch나 commit만 가져오도록 인자를 줄 수 있다. <source>:<destination> 형태를 사용하면 원격의 특정 위치를 가져와 로컬 branch에 저장할 수 있다. 

git fetch origin foo
git fetch origin C3:foo
git fetch origin C6:main

source가 없는 경우

source 부분을 비우면 특수한 동작을 한다. git push origin :foo는 원격 저장소의 foo branch를 삭제한다. git fetch origin :bar는 Learn Git Branching 학습 환경에서는 비어 있는 source를 이용해 로컬 bar branch를 만드는 예시로 사용된다. 

git push origin :foo
git fetch origin :bar

pull의 인자

git pull도 fetch와 merge가 합쳐진 명령어이기 때문에 인자를 줄 수 있다. git pull origin foo는 원격의 foo를 가져온 뒤 현재 branch에 병합하는 흐름이고, git pull origin bar:bugFix는 원격 bar를 로컬 bugFix로 가져온 뒤 bugFix를 현재 branch에 병합하는 흐름이다. 

git pull origin foo
# git fetch origin foo + git merge FETCH_HEAD

git pull origin bar:bugFix
# git fetch origin bar:bugFix + git merge bugFix

git pull origin main
git pull origin main:foo

마무리

Git은 처음엔 명령어가 많고 복잡해 보여 막막했지만, commit 그래프와 branch의 흐름을 이해하고 나니 각 명령어가 왜 필요한지 자연스럽게 연결되기 시작했다. 단순히 명령어를 외우는 것이 아니라 Git이 어떻게 동작하는지를 직접 눈으로 확인하며 익힌 점이 가장 큰 수확이었다.

앞으로 프로젝트나 협업 과정에서 Git을 사용할 때도 무작정 명령어를 입력하기보다, 현재 어떤 위치에 있고 어떤 변화가 일어날지 생각하면서 활용하는 습관을 들이고 싶다.
